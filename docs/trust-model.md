# Trust Model & Developer Protocols

> Section 1 of the Karsheft Flock Specification | Version 0.1.0-draft

---

## 1.1 Overview

Karsheft Flock uses a **tiered trust model**. Trust is established once, at vendor time in CI/CD, and
propagated to all install contexts via a version-controlled lockfile. The mirror is never trusted at
install time — only the lockfile is.

```
Trust anchor: CI/CD environment
     │
     │  GPG verify against upstream signing key (vendor time only)
     │  SHA256 of each .deb captured into flock.lock
     │  flock.lock committed to version control
     │
     ├── Developer install path
     │     fetch .deb from mirror
     │     SHA256 verify against flock.lock
     │     dpkg install
     │     (mirror trust assumed; fast path)
     │
     └── End-user / release install path (CI/CD)
           fetch .deb from mirror
           GPG verify against pinned key
           SHA256 verify against flock.lock
           dpkg install
           (full verification; same lockfile; identical result)
```

---

## 1.2 The Lockfile as Source of Truth

`flock.lock` is the single source of truth for all install contexts. It is:

- Committed to version control alongside application code
- Generated by `flock resolve` in CI/CD with `--verify=full`
- Read-only at install time — never modified by `flock install`
- Human-readable and diff-friendly (TOML format)

The lockfile records the SHA256 of the `.deb` file itself (before extraction), not of
installed files. This means verification happens before dpkg executes any maintainer
scripts, closing the window where a tampered package could run code before detection.

---

## 1.3 Verification Levels

| Level | Flag | Used by | Checks |
|---|---|---|---|
| `checksum` | `--verify=checksum` | Developers (default) | SHA256 of .deb vs flock.lock |
| `full` | `--verify=full` | CI/CD, release builds | GPG signature + SHA256 of .deb |
| `none` | `--verify=none` | Air-gapped/emergency | No verification (logged, warned) |

`--verify=none` is explicitly logged as a security downgrade in the audit trail and
requires a confirmation flag `--i-understand-the-risk` to proceed.

## 1.4 Developer Trust Protocols

### 1.4.1 When checksum verification is acceptable

A developer may use `--verify=checksum` (the default) when ALL of the following are true:

- The developer is on a network they control or trust (corporate VPN, home network)
- The `flock.lock` file was generated by a CI/CD run with `--verify=full`
- The `flock.lock` commit is signed (recommended: require signed commits on main)
- The developer has pulled the latest `flock.lock` from the canonical branch

### 1.4.2 When developers MUST use `--verify=full`

- On untrusted networks (conference wifi, public hotspots)
- When adding or upgrading packages (running `flock resolve`)
- When the `flock.lock` has been modified locally and not yet validated by CI/CD
- When working with packages from a non-official mirror

### 1.4.3 Developer protocol for adding a new package

```
1. Run: flock resolve --pkg <name> --verify=full
2. Review flock.lock diff: confirm version, SHA256, and GPG fingerprint
3. Commit flock.lock alongside the code change that requires the new package
4. CI/CD re-runs flock resolve --verify=full and asserts lockfile is unchanged
5. PR is merged; all developers pull the new flock.lock before running flock install
```

### 1.4.4 Developer protocol for upgrading a package

```
1. Run: flock resolve --pkg <name> --verify=full
2. Review flock.lock diff: confirm only the target package changed
3. Check upstream changelog for the new version
4. Commit flock.lock with a message referencing the version change
5. CI/CD validates the new lockfile before merge
```

## 1.5 Trust Hierarchy Summary

| Tier       | SHA256 | GPG | Risk Flag Required | Recommended For         |
|------------|--------|-----|--------------------|-------------------------|
| `checksum` | ✅     | ❌  | No                 | Developer installs      |
| `full`     | ✅     | ✅  | No                 | CI/CD, production       |
| `none`     | ❌     | ❌  | **Yes**            | Emergency only          |

---

## 1.6 Security Recommendations

1. **Commit `flock.lock` to version control.** It is a security artifact, not a build artifact.
2. **Generate `flock.lock` with `--verify=full` in CI/CD.** Developers install from it with `--verify=checksum`.
3. **Require signed commits on the branch that owns `flock.lock`.** This closes the supply chain gap between CI/CD generation and developer consumption.
4. **Pin your mirror URL.** Changing mirrors mid-project can silently introduce different packages.
5. **Rotate GPG keys carefully.** Re-run `flock resolve --verify=full` after a key rotation and review the full lockfile diff.
6. **Treat `--i-understand-the-risk` as an alarm.** Any use of this flag must be justified and trigger a post-incident security review.
